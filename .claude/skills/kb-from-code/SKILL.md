# /kb-from-code — черновик базы знаний из кода

Ты читаешь исходники целевого продукта и пишешь **черновик** Q&A-базы для onbo
**на языке конечного пользователя** — так, как формулировал бы саппорт, а не
разработчик. Результат — всегда черновик под ревью: юзер правит его в веб-админке
`/admin` перед тем, как он попадёт в прод. Скриптов нет — всю работу делаешь ты.

## Вход

- **Путь к коду** целевого продукта (обязательно).
- **Язык базы знаний** — опционально, по умолчанию русский.
- **Коллекция** — опционально, по умолчанию `common`.
- **Доп. языки** — опционально: список языков для мультиязычного сайта (см. «Переводы»).

## Шаг 1. Разведка (для больших репо — по приоритету)

Не читай весь репозиторий. Иди по источникам в порядке ценности и останавливайся,
когда собран инвентарь фич:

1. **README / доки** — общая картина, глоссарий, названия фич.
2. **Роуты / контроллеры** — какие вообще есть разделы и операции.
3. **UI-страницы настроек / профиля / онбординга** — что пользователь реально делает руками.
4. **i18n-файлы** (`locales/`, `*.po`, `messages.*.json`, `en.yaml` и т.п.) —
   **золотой источник** пользовательских названий кнопок, пунктов меню и разделов.
   Именно отсюда берутся точные формулировки путей в UI.
5. **Шаблоны / навигация** — как разделы называются и вложены друг в друга.

Составь короткий **инвентарь фич**: что пользователь может настроить/сделать, и как
эти разделы называются в интерфейсе (по i18n/шаблонам).

## Шаг 2. Правила формулирования

Q&A пишутся **для новичка-пользователя**, а не для инженера.

- **Только язык пользователя.** Запрещены: HTTP-методы, пути API (`POST /api/...`),
  имена функций, таблиц, переменных, классов.
- **Путь в UI восстанавливай из i18n/шаблонов.** Если путь недостоверен — пиши
  обобщённо и добавляй строкой `# TODO: проверить путь в UI`.
- **Один вопрос = одна задача новичка.** Не склеивай несколько задач в одну пару.
- **Роли/гварды в коде → `department`/`roles`.** Если раздел за проверкой роли
  (`@requires_role("accountant")`, `if user.is_staff` и т.п.) — проставь аудиторию
  у пары; иначе оставь пустой (доступно всем).

Плохо / хорошо:

| ❌ Плохо (язык кода) | ✅ Хорошо (язык пользователя) |
|---|---|
| «Отправьте `POST /api/users/{id}/language` с полем `language`» | «Напишите ассистенту “смени язык на английский” — язык переключится сразу» |
| «Вызовите `reset_password_token()`» | «Откройте Настройки → Безопасность и нажмите «Сменить пароль»» |
| «Смена email и телефона через `UserSettingsView`» | Две отдельные пары: «Как сменить email?» и «Как сменить телефон?» |

## Шаг 3. Выход

Запиши черновик в `tmp/kb-from-code/<проект>/draft_faq.yaml` (временные файлы —
только в `tmp/` проекта, никогда в `/tmp`). Формат — как `config/seed_faq.yaml`
(поле `video_url` НЕ добавляй — это забота скилла `/kb-video`):

```yaml
qa:
  - question: "..."
    answer: "..."
    collection: common        # опц., деф. common
    department: accounting     # опц., если раздел за ролью/отделом
    roles: [accountant]        # опц.
```

Целевой объём — **15–40 пар**. Покажи юзеру короткую сводку: сколько пар, по каким
разделам, какие помечены аудиторией, где стоят `# TODO`.

## Шаг 4. Импорт (по подтверждению)

Только после «да» от юзера:

```bash
onbo kb import tmp/kb-from-code/<проект>/draft_faq.yaml
```

Затем предложи открыть `/admin` и вычитать/поправить пары.

**Дедупликация:** если веб-канал onbo запущен — сначала `GET /admin/api/qa`,
сверься с уже существующими вопросами. `add_qa` идемпотентен по паре
`collection + question`, так что повторный импорт не плодит дубли, но лучше не
предлагать пары, которые уже есть.

## Переводы (мультиязычные сайты)

Если юзер назвал доп. языки — сгенерируй параллельные пары на каждом языке
**с учётом культурных особенностей носителей** (не дословный перевод, а адаптация
формулировок), той же коллекции/аудитории. Клади в тот же `draft_faq.yaml`.

## Edge cases

- **API-only продукт** (нет UI) — спроси юзера, как выглядит клиент/интерфейс;
  без UI писать «Настройки → …» нельзя.
- **Не-русский продукт** — уточни язык базы знаний (не угадывай по коду).
- **Монорепо** — уточни, какой пакет/сервис онбордим.

## Проверка

Прогони на `onbo/demo` или соседнем проекте: в YAML нет API-терминов, `onbo kb import`
отрабатывает, ассистент отвечает из новой пары в `/chat`.
