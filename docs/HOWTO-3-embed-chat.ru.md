# How-to 3. Встраивание чата в свой продукт: пользователи, роли, виджет

*[English](HOWTO-3-embed-chat.md) · Русский*

Продолжение [HOWTO-2-kb-and-chat.ru.md](HOWTO-2-kb-and-chat.ru.md): стенд поднят,
в базе знаний уже есть содержимое. Дальше — поставить окно чата внутрь своего
продукта, чтобы человек, залогиненный в *ваше* приложение, общался с onbo от
своего имени и видел только то, что позволено его отделу и ролям.

Нужно сделать две вещи, и по отдельности они не работают:

1. **Перенести пользователей.** onbo решает, что человеку можно показывать, по
   своей таблице `app_user`, а не по вашей сессии. Если таблица пустая — все
   чужие и видят только общедоступное.
2. **Поставить свой бэкенд между браузером и onbo.** У `POST /chat` нет
   собственной авторизации: он доверяет тому `user_id`, который пришёл в теле
   запроса.

---

## 1. Общая схема

```
браузер (виджет)  →  ВАШ бэкенд /api/assistant  →  onbo POST /chat
   кука сессии         подставляет user_id           порт 18000, внутренняя сеть
                       из сессии
```

Браузер никогда не ходит в onbo напрямую. Ваш бэкенд и так знает, кто залогинен,
и он — единственное место, которое имеет право сказать «этот запрос от u_1042».

**Что будет без прокси:** любой откроет консоль и отправит
`{"user_id": "ceo@acme.com", "text": "..."}`. onbo спокойно ответит материалами
директора, потому что с его точки зрения это и есть директор. Именно эта ошибка
превращает всё описанное ниже разграничение доступа в украшение.

## 2. Перенос пользователей и ролей

### 2.1. Что хранит onbo

По одной строке на человека в таблице `app_user` в Postgres:

| поле         | что это                                                       |
|--------------|---------------------------------------------------------------|
| `user_id`    | id, которым пользуется ваш продукт. Что угодно стабильное и уникальное: внутренний id, почта, telegram id. |
| `department` | один отдел, например `accounting`. Необязательно.             |
| `roles`      | список, например `[accountant, admin]`. Необязательно.        |

Это весь профиль. Поиск строит из этих двух полей фильтр видимости: человек
видит материалы своего отдела, материалы своих ролей и общие (без тегов). Фильтр
строится только по этой таблице — не по тексту вопроса и не по модели, поэтому
«покажи документы соседнего отдела» не может сработать.

Тот, кого в таблице **нет**, получает минимальные права: без отдела,
`roles: ["employee"]`, только общедоступное. Ничего не ломается — просто видно
меньше.

### 2.2. Названия отделов и ролей

Одни и те же названия используются здесь и в тегах базы знаний — они
сравниваются как обычные строки. Определитесь один раз:

```bash
# база знаний: кому можно видеть материал
docker compose exec app onbo kb add-doc docs/payroll.md \
  --collection accounting --department accounting --roles accountant

# справочник: кто этот человек
docker compose exec app onbo users add u_1042 \
  --department accounting --roles accountant
```

Опечатка (`accountants` вместо `accountant`) нигде не вызовет ошибки — материал
просто молча перестанет находиться. Держите список названий коротким и
записанным.

### 2.3. Массовый импорт

Выгрузите свой справочник в YAML-файл со списком `users:` на верхнем уровне —
формат расписан в `config/users.example.yaml`:

```yaml
users:
  - user_id: "u_1042"
    department: accounting
    roles: [accountant]

  - user_id: "alice@example.com"
    department: support
    roles: [support, admin]
```

```bash
docker compose exec app onbo users import ./my_users.yaml
```

Импорт идемпотентный: повторный запуск перезаписывает отдел и роли уже
существующих и добавляет остальных. Импорт никого не удаляет — уволенный
сотрудник сохранит старый профиль, пока вы не почистите его сами.

### 2.4. Как поддерживать в актуальном состоянии

Выберите по тому, как часто у вас меняется оргструктура:

- **Руками** — `onbo users add` при выходе нового человека или смене команды.
  Для небольшой команды этого достаточно.
- **По расписанию** — крон на вашей стороне выгружает справочник в
  `my_users.yaml` и запускает `onbo users import`. Обычный вариант.
- **При логине** — ваш бэкенд сам пишет строку в ту же базу Postgres при
  создании сессии. Таблица обычная, ничего специфичного для onbo:

  ```sql
  INSERT INTO app_user (user_id, department, roles) VALUES ($1, $2, $3)
  ON CONFLICT (user_id) DO UPDATE
    SET department = EXCLUDED.department, roles = EXCLUDED.roles;
  ```

  `roles` — JSON-колонка, передавайте JSON-массив. Поле `welcomed_at` не
  трогайте: по нему onbo понимает, показывал ли он уже приветственную сводку.

HTTP-эндпоинта для справочника сейчас нет — только CLI или SQL.

## 3. Прокси-эндпоинт на вашем бэкенде

Один эндпоинт, без состояния. Express:

```js
app.post("/api/assistant", async (req, res) => {
  const user = req.session.user;               // your own auth, not onbo's
  if (!user) return res.sendStatus(401);

  const upstream = await fetch("http://app:18000/chat", {
    method: "POST",
    headers: { "content-type": "application/json" },
    body: JSON.stringify({
      user_id: user.id,                        // NEVER take this from the body
      text: String(req.body.text || "").slice(0, 4000),
      locale: "ru",
    }),
  });
  res.status(upstream.status).json(await upstream.json());
});
```

FastAPI:

```python
@app.post("/api/assistant")
async def assistant(body: dict, user=Depends(current_user)):
    async with httpx.AsyncClient(timeout=60) as client:
        upstream = await client.post(
            "http://app:18000/chat",
            json={"user_id": user.id, "text": body["text"][:4000], "locale": "ru"},
        )
    return upstream.json()
```

Что важно:

- `http://app:18000` — имя сервиса из `docker-compose.yml`. Если ваш бэкенд
  живёт в другом месте, укажите адрес, до которого он реально дотягивается.
- Ставьте запас по таймауту (30–60 с). Холодная модель плюс поиск — это дольше
  обычного вызова API.
- Если пользуетесь действиями в режиме `confirm`, проксируйте так же
  `POST /confirm` (`{user_id, action, approved}`) — и там тоже подставляйте
  `user_id` сами.
- Массив `results` из ответа можно отдавать в браузер как есть: в нём те же
  материалы, которые человеку и так разрешено видеть.

### Ответ

```json
{
  "text": "…текст, который читает пользователь…",
  "results": [{"type": "rag", "status": "ok", "…": "…"}],
  "welcomed": false
}
```

`welcomed: true` означает, что это было первое сообщение человека и к `text`
спереди приклеена одноразовая приветственная сводка. Показывайте `text` и, по
желанию, рисуйте карточки для результатов со `status: "needs_confirm"`.

## 4. Виджет в браузере

`docs/examples/chat.html` — рабочий чат на 160 строк: текст, голос и карточки
Ок/Отмена для действий в режиме confirm. Это **демо-страница**, а не виджет: она
спрашивает `user_id` в поле ввода. Чтобы переиспользовать её внутри продукта:

1. Удалите поле `user_id` и строчку про демо-пользователей в шапке, а из скрипта
   — все `user_id: userInput.value`: его подставит прокси.
2. Переведите запросы на свой эндпоинт: `post('/chat', …)` → `post('/api/assistant', …)`,
   `post('/confirm', …)` → ваш прокси для подтверждений, а стартовый вызов
   `post('/welcome', …)` либо заведите через третий маршрут, либо просто уберите
   (сводка всё равно придёт с первым сообщением).
3. Добавьте `credentials: 'same-origin'` в опции `fetch`, если этого требует ваша
   кука сессии.
4. Вставьте разметку в свою страницу либо отдавайте файл со своего домена внутри
   `<iframe>`.

Голос (`POST /voice`) — это `multipart/form-data`, то есть проксировать придётся
файл, а не JSON. Если голос во встроенном виджете не нужен, уберите кнопку
микрофона — это меньшая половина страницы.

## 5. Закрыть onbo снаружи

Пока стенд доступен только с вашей машины, опубликованные порты удобны. До того,
как к нему придут живые пользователи:

- Уберите `ports: - "18000:18000"` у сервиса `app`, если ваш бэкенд в той же
  docker-сети: `http://app:18000` продолжит работать, а снаружи попасть будет
  некуда. Иначе закройте onbo правилом фаервола, пускающим только адрес вашего
  бэкенда.
- Задайте `ONBO_ADMIN_TOKEN` в `.env`. Без него `/admin` открыт любому, кто
  дотянулся до порта, а оттуда правится база знаний.
- То же про `demo-backend` (порт 18100) — уберите сервис совсем, когда действия
  будут ходить в настоящий бэкенд через `PRODUCT_API_BASE`.

## 6. Чек-лист перед показом людям

- [ ] `onbo users import` отработал, `onbo kb status` показывает содержимое.
- [ ] Названия отделов и ролей в справочнике совпадают с тегами на материалах.
- [ ] Браузер ходит в **ваш** эндпоинт, `user_id` берётся из сессии.
- [ ] Разлогинились → эндпоинт отдаёт 401, а не общедоступный ответ.
- [ ] Зашли под человеком из другого отдела → его материалов в ответе нет.
      Проверьте настоящим аккаунтом, а не рассуждением.
- [ ] Порт 18000 недоступен снаружи, `ONBO_ADMIN_TOKEN` задан.
