# How-to 3. Встраивание чата в свой продукт: пользователи, роли, виджет

*[English](HOWTO-3-embed-chat.md) · Русский*

Продолжение [HOWTO-2-kb-and-chat.ru.md](HOWTO-2-kb-and-chat.ru.md): стенд поднят,
в базе знаний уже есть содержимое. Дальше — поставить окно чата внутрь своего
продукта, чтобы человек, залогиненный в *ваше* приложение, общался с onbo от
своего имени и видел только то, что позволено его отделу и ролям.

---

## 1. Единственное правило

У `POST /chat` нет собственного входа. То, что сообщает ему, кто спрашивает,
определяет и то, что этому человеку можно показать, — поэтому **оно не должно
приходить из браузера без защиты**. Иначе любой откроет консоль, отправит
`{"user_id": "ceo@acme.com", "text": "…"}` и прочитает материалы гендиректора.

Выполнить это правило можно двумя безопасными способами. Выберите один:

| | **A. Прокси** | **B. Подписанный токен** |
|---|---|---|
| Кто знает роли | onbo, из своей таблицы `app_user` | ваш бэкенд, внутри токена |
| Синхронизация пользователей | нужна | **не нужна** |
| Браузер обращается | к вашему бэкенду | напрямую к onbo |
| Что настроить | одну ручку | общий секрет и несколько строк подписи |
| Кому подходит | несколько сотен человек, редкие изменения | десятки тысяч, постоянные приходы и уходы |

Режим B появился именно потому, что копировать справочник пользователей в onbo
перестаёт быть разумным на масштабе: миллион человек, тысяча пришла и тысяча
ушла за день — это отдельная работа по синхронизации, которую никто не хочет
вести. В режиме B onbo не хранит пользователей вообще: профиль приезжает с
каждым запросом, подписанный.

---

## 2. Режим A — запросы идут через ваш бэкенд

```
браузер (виджет)  →  ВАШ бэкенд /api/assistant  →  onbo POST /chat
   cookie сессии       подставляет user_id           порт 18000, закрытая сеть
```

### 2.1. Что onbo хранит о человеке

Одна строка на человека в таблице `app_user` в Postgres:

| поле         | смысл                                                        |
|--------------|--------------------------------------------------------------|
| `user_id`    | идентификатор из вашего продукта. Любой стабильный и уникальный: внутренний id, почта, telegram id. |
| `department` | один отдел, например `accounting`. Необязательно.            |
| `roles`      | список, например `[accountant, admin]`. Необязательно.       |

Это весь профиль. Из этих двух полей строится фильтр видимости: человек видит
материалы, помеченные его отделом, материалы, помеченные одной из его ролей, и
непомеченные (общие). Фильтр строится только по этой таблице — никогда по тексту
вопроса и никогда моделью, поэтому «покажи документы соседнего отдела» не
сработает.

Кого в таблице нет, получает минимальные права: без отдела,
`roles: ["employee"]`, только публичное. Ничего не ломается — просто видно
меньше.

### 2.2. Названия отделов и ролей

Здесь и в метках базы знаний должны быть одни и те же названия — они
сравниваются как обычные строки. Договоритесь один раз:

```bash
# база знаний: кому можно этот материал
docker compose exec app onbo kb add-doc docs/payroll.md \
  --collection accounting --department accounting --roles accountant

# справочник: кто этот человек
docker compose exec app onbo users add u_1042 \
  --department accounting --roles accountant
```

Опечатка (`accountants` вместо `accountant`) нигде не вызовет ошибку — она
просто молча спрячет материал. Держите список названий коротким и записанным.

### 2.3. Массовый импорт

Выгрузите справочник в YAML-файл со списком `users:` — формат описан в
`config/users.example.yaml`:

```yaml
users:
  - user_id: "u_1042"
    department: accounting
    roles: [accountant]

  - user_id: "alice@example.com"
    department: support
    roles: [support, admin]
```

```bash
docker compose exec app onbo users import ./my_users.yaml
```

Импорт идемпотентен: повторный запуск перезаписывает отдел и роли тех, кто в
файле, и добавляет остальных. Никто никогда не удаляется импортом — человек,
убранный из вашей кадровой системы, сохраняет старый профиль, пока вы его не
почистите.

### 2.4. Как поддерживать в актуальном состоянии

Выберите по частоте изменений:

- **Руками** — `onbo users add` при выходе нового человека или смене команды.
  Нормально для небольшой команды.
- **По расписанию** — ваш cron выгружает справочник в `my_users.yaml` и
  запускает `onbo users import`. Обычный вариант.
- **При входе** — ваш бэкенд пишет строку в ту же базу Postgres при создании
  сессии. Таблица — обычный SQL, ничего специфичного для onbo:

  ```sql
  INSERT INTO app_user (user_id, department, roles) VALUES ($1, $2, $3)
  ON CONFLICT (user_id) DO UPDATE
    SET department = EXCLUDED.department, roles = EXCLUDED.roles;
  ```

  `roles` — JSON-колонка, передавайте JSON-массив. `welcomed_at` не трогайте: по
  нему onbo понимает, показывал ли уже приветствие.

Если поддержание всего этого в актуальном виде звучит как лишняя работа — именно
её убирает режим B.

### 2.5. Ручка-прокси

Одна ручка, без состояния. Express:

```js
app.post("/api/assistant", async (req, res) => {
  const user = req.session.user;               // ваша авторизация, не onbo
  if (!user) return res.sendStatus(401);

  const upstream = await fetch("http://app:18000/chat", {
    method: "POST",
    headers: { "content-type": "application/json" },
    body: JSON.stringify({
      user_id: user.id,                        // НИКОГДА не берите из тела запроса
      text: String(req.body.text || "").slice(0, 4000),
      locale: "ru",
    }),
  });
  res.status(upstream.status).json(await upstream.json());
});
```

FastAPI:

```python
@app.post("/api/assistant")
async def assistant(body: dict, user=Depends(current_user)):
    async with httpx.AsyncClient(timeout=60) as client:
        upstream = await client.post(
            "http://app:18000/chat",
            json={"user_id": user.id, "text": body["text"][:4000], "locale": "ru"},
        )
    return upstream.json()
```

Что важно:

- `http://app:18000` — имя сервиса из `docker-compose.yml`. Если ваш бэкенд
  живёт в другом месте, укажите адрес, до которого он реально достаёт.
- Ставьте щедрый таймаут (30–60 с). Холодная модель плюс поиск медленнее
  обычного API-вызова.
- Если используете действия с `mode: confirm`, проксируйте так же
  `POST /confirm` (`{user_id, action, approved}`) — снова подставляя `user_id`
  сами.
- Виджет ждёт, что ручка подтверждения лежит рядом с ручкой чата: если чат —
  `/api/assistant/chat`, то подтверждение — `/api/assistant/confirm`. Иначе
  укажите `confirmEndpoint` явно.

## 3. Режим B — подписанный токен, без справочника

Ваш бэкенд и так знает, кто пришёл. Вместо того чтобы копировать это знание в
onbo, он выдаёт браузеру короткоживущий **подписанный токен** с идентификатором,
отделом и ролями внутри; браузер отправляет его в onbo с каждым вопросом, а onbo
верит содержимому, потому что подпись невозможно подделать.

Токен не зашифрован, и шифровать его не нужно — его ценность в том, что он
*неподделываемый*. Правка `"roles": ["intern"]` на `"roles": ["admin"]` ломает
подпись, и onbo отвечает 401.

### 3.1. Общий секрет

```bash
openssl rand -hex 32          # обе стороны используют одну и ту же строку
```

`.env`:

```ini
ONBO_JWT_SECRET=<строка выше>
ONBO_ALLOW_USER_ID=false                    # в бою: только подписанный токен
ONBO_CORS_ORIGINS=https://app.example.com   # сайт, на котором стоит виджет
```

```bash
docker compose up -d app
```

`ONBO_ALLOW_USER_ID=false` закрывает старую дверь: как только ручка доступна из
браузеров, голый `user_id` в теле запроса ничего не доказывает — его перестают
принимать. `ONBO_CORS_ORIGINS` перечисляет ровно те адреса, чьим страницам можно
обращаться к onbo: схема, хост и порт должны совпадать. `*` без секрета
отвергается при старте, а не разрешается молча.

### 3.2. Выдача токена на вашей стороне

Это обычный JWT, HS256. Совместимый токен сделает любая библиотека.

Node (`jsonwebtoken`):

```js
app.get("/api/assistant/token", (req, res) => {
  const user = req.session.user;
  if (!user) return res.sendStatus(401);
  res.json({
    token: jwt.sign(
      { sub: user.id, department: user.department, roles: user.roles },
      process.env.ONBO_JWT_SECRET,
      { expiresIn: "10m" },
    ),
  });
});
```

Python (`pyjwt`):

```python
@app.get("/api/assistant/token")
async def assistant_token(user=Depends(current_user)):
    claims = {
        "sub": user.id,
        "department": user.department,
        "roles": user.roles,
        "exp": int(time.time()) + 600,
    }
    return {"token": jwt.encode(claims, os.environ["ONBO_JWT_SECRET"], algorithm="HS256")}
```

Поля токена:

| поле | смысл |
|---|---|
| `sub` | идентификатор пользователя. Обязателен. По нему живут сессия и одноразовое приветствие. |
| `department` (или `dept`) | один отдел. Необязательно. |
| `roles` | список названий или идентификаторов ролей — как устроено у вас. Одиночное значение без списка тоже принимается. |
| `exp` | срок годности, unix-секунды. **Обязателен** — токен без срока это постоянный ключ. |

Держите срок коротким (5–15 минут) и позвольте виджету запрашивать новый:
`getToken()` вызывается перед каждым запросом, так что свежий токен стоит одного
кэшируемого вызова на вашей стороне.

Названия ролей должны точно совпадать с метками на материалах базы знаний — то
же правило, что в §2.2, и та же молчаливая осечка, если не совпадут.

### 3.3. Проверить руками

```bash
docker compose exec app onbo token u_1042 --department accounting --roles accountant
# → eyJhbGciOiJIUzI1NiIs…

curl -s -X POST http://localhost:18000/chat -H 'Content-Type: application/json' \
  -d '{"token":"eyJ…","text":"сколько у меня дней отпуска?"}'
```

Поменяйте один символ в середине токена — тот же запрос вернёт 401. В этом весь
механизм защиты, и на это стоит посмотреть один раз.

## 4. Виджет

`docs/examples/onbo-widget.js` — самодостаточное окно чата: кнопка-пузырь,
панель, ссылки кнопками, карточки Ок/Отмена для действий с подтверждением,
опционально голос. Без зависимостей и без сборки. Рисуется внутри shadow root,
поэтому CSS вашего сайта не протечёт внутрь, а его собственный — наружу.

**Идентификатор пользователя он не отправляет никогда.** Либо его подставляет
ваш прокси (режим A), либо он лежит в токене (режим B).

Скопируйте файл туда, откуда ваш сайт отдаёт статику:

```bash
cp docs/examples/onbo-widget.js public/
```

### 4.1. Статический HTML — вообще без кода

```html
<script type="module" src="/onbo-widget.js"
        data-endpoint="/api/assistant/chat"
        data-title="Помощник"></script>
```

Или с настройками — это то же самое, только расписанное:

```html
<script type="module">
  import { init } from '/onbo-widget.js';
  init({ endpoint: '/api/assistant/chat', title: 'Помощник' });
</script>
```

### 4.2. Режим токена

`endpoint` смотрит прямо в onbo, а `getToken` берёт свежий токен у вашего
бэкенда:

```js
init({
  endpoint: 'https://onbo.example.com/chat',
  getToken: async () => (await fetch('/api/assistant/token', {
    credentials: 'same-origin',
  }).then(r => r.json())).token,
});
```

Именно этому варианту нужен `ONBO_CORS_ORIGINS` — браузер общается с onbo
напрямую.

### 4.3. React

```jsx
import { useEffect, useRef } from 'react';
import { init } from './onbo-widget.js';

export function Assistant() {
  const widget = useRef(null);
  useEffect(() => {
    widget.current = init({ endpoint: '/api/assistant/chat' });
    return () => widget.current.destroy();   // переживает двойной монтаж StrictMode
  }, []);
  return null;   // виджет сам цепляется к <body>
}
```

### 4.4. Vue

```vue
<script setup>
import { onMounted, onUnmounted } from 'vue';
import { init } from './onbo-widget.js';

let widget;
onMounted(() => { widget = init({ endpoint: '/api/assistant/chat' }) });
onUnmounted(() => widget.destroy());
</script>
```

### 4.5. Nuxt 3 — целиком, файлами

`docs/examples/nuxt/` — готовая интеграция в режиме A: четыре серверные ручки
(чат, подтверждение, приветствие, голос), плагин, который показывает окно
залогиненным, и инструкция, что скопировать и что поправить. Править нужно ровно
две вещи: как достать пользователя из сессии и адрес onbo.

```bash
cp -r docs/examples/nuxt/server docs/examples/nuxt/plugins ~/my-nuxt-app/
cp docs/examples/onbo-widget.js ~/my-nuxt-app/public/
```

Подробности — в [docs/examples/nuxt/README.md](examples/nuxt/README.md).

### 4.6. Остальные фреймворки

Angular, Svelte и остальные — по той же схеме: `init()`, когда компонент
появился, `destroy()`, когда исчез. Если ваш сборщик не может импортировать
файл, подключите его тегом `<script type="module">` и вызывайте
`window.OnboWidget.init(...)`.

### 4.7. Настройки

| настройка | по умолчанию | смысл |
|---|---|---|
| `endpoint` | `/api/assistant` | куда уходят вопросы: ваша ручка-прокси или `/chat` самого onbo |
| `confirmEndpoint` | выводится | ручка подтверждения; по умолчанию соседняя с `endpoint` |
| `welcomeEndpoint` | выводится | ручка приветствия, то же правило |
| `voiceEndpoint` | `null` | загрузка голоса (multipart). Не задано — кнопки микрофона нет |
| `getToken` | `null` | `() => токен`, синхронно или асинхронно. Режим B |
| `headers` | `null` | дополнительные заголовки (например CSRF); объект или функция |
| `credentials` | `same-origin` | `include`, если прокси на другом домене |
| `title` / `subtitle` | `Помощник` | шапка панели |
| `accent` | `#2f6feb` | единственный цвет, из которого собрано остальное |
| `position` | `right` | `right` или `left` |
| `theme` | `auto` | `auto` следует за системной темой посетителя |
| `locale` | `ru` | передаётся в onbo; он же язык распознавания речи |
| `open` | `false` | открыть панель сразу |
| `greetOnOpen` | `true` | запросить приветствие при первом открытии |
| `launcher` | `true` | `false` убирает пузырь — открывайте своей кнопкой |
| `mount` | `<body>` | элемент, к которому прицепиться |
| `zIndex` | очень большой | поднимите, если ваши оверлеи оказались выше |
| `strings` | русские | все видимые подписи, можно переопределить любую часть |

`init()` возвращает `{ open, close, toggle, ask, isOpen, destroy }` — так что
ваша ссылка «Помощь» может сделать `widget.ask('Как сменить почту?')`: панель
откроется, и вопрос уйдёт.

### 4.8. Голос

Голос передаётся как `multipart/form-data`, поэтому в режиме A нужна ручка,
пробрасывающая файл, а не JSON. В режиме B укажите `voiceEndpoint` прямо на
`/voice` onbo. Микрофон браузеры дают только в защищённом контексте:
`localhost` считается, удалённому хосту нужен HTTPS.

### 4.9. Что приходит в ответ

```json
{
  "text": "…ответ, который читает человек…",
  "results": [{"type": "rag", "status": "ok", "links": [{"title": "Мои проекты", "url": "…"}]}],
  "welcomed": false
}
```

`welcomed: true` означает, что это было первое сообщение человека и к `text`
добавлено одноразовое приветствие. `results[].links` — места в продукте, на
которые указывает ответ; те же ссылки продублированы в конце `text` обычным
блоком под заголовком `Ссылки:`, поэтому клиент, умеющий только текст, ничего не
теряет. Виджет рисует структурные ссылки кнопками, а дублирующий блок из текста
срезает — это стоит знать, если пишете свой интерфейс.

`docs/examples/chat.html` остаётся простой демо-страницей (там `user_id`
вводится руками, так что это локальная проверка, а не то, что встраивают).
Отдавать её надо из `media/` — пути внутри относительные, поэтому открытый с
диска файл ломает все запросы:

```bash
cp docs/examples/chat.html media/
# открыть http://localhost:18000/media/chat.html
```

## 5. Закрываем onbo снаружи

Пока стенд доступен только с вашей машины, опубликованные порты удобны. До того
как придут настоящие пользователи:

- **Режим A:** уберите `ports: - "18000:18000"` у сервиса `app`, если ваш бэкенд
  в той же docker-сети — `http://app:18000` продолжит работать, а снаружи входа
  не станет. Иначе закройте onbo правилом фаервола, пускающим только адрес
  вашего бэкенда.
- **Режим B:** порт обязан быть доступен браузерам, поэтому вместо этого
  поставьте onbo за TLS на своём домене и опирайтесь на
  `ONBO_ALLOW_USER_ID=false` плюс точный `ONBO_CORS_ORIGINS`. Никогда не `*`.
- Задайте `ONBO_ADMIN_TOKEN` в `.env`. Без него `/admin` открыт любому, кто
  дотянется до порта, а оттуда правится база знаний.
- То же про `demo-backend` (порт 18100) — удалите сервис целиком, когда действия
  начнут ходить в настоящий бэкенд через `PRODUCT_API_BASE`.

## 6. Чек-лист перед показом людям

- [ ] `onbo kb status` показывает содержимое, а названия отделов и ролей в нём
      совпадают с теми, что использует ваш продукт.
- [ ] Режим A: `onbo users import` отработал; браузер обращается к **вашей**
      ручке, а `user_id` берётся из сессии.
- [ ] Режим B: `ONBO_ALLOW_USER_ID=false`, в `ONBO_CORS_ORIGINS` перечислен
      ровно ваш сайт, срок жизни токенов — минуты.
- [ ] Вышли из аккаунта → ручка отвечает 401, а не публичным ответом.
- [ ] Вошли под человеком из другого отдела → его материалов в ответе нет.
      Проверьте настоящим аккаунтом, а не рассуждением.
- [ ] Токен или user_id, поправленный руками, получает 401.
- [ ] `ONBO_ADMIN_TOKEN` задан, и `/admin` его спрашивает.
