# How-to 1. Запустить и настроить onbo в своём проекте

*[English](HOWTO-1-setup.md) · Русский*

Про наполнение базы знаний, команды и чат — во втором файле:
[HOWTO-2-kb-and-chat.ru.md](HOWTO-2-kb-and-chat.ru.md).

Что вы получите после этого файла: поднятый ассистент на `http://localhost:18000`,
подключённую модель (локальную или облачную) и настроенную связку со своим
продуктом.

---

## 1. Что нужно на машине

- Docker + Docker Compose (всё приложение живёт в контейнерах, ничего ставить в
  систему не надо);
- ~6 ГБ диска: образы, веса эмбеддингов (~1 ГБ) и модель распознавания речи;
- ключ API OpenAI — в конфигурации по умолчанию он нужен и чат-модели, и
  эмбеддингам. Провайдера можно заменить или перейти на полностью локальные
  модели (разделы 3 и 4).

## 2. Запуск за три команды

```bash
git clone git@github.com:japanes/onbo.git && cd onbo
cp .env.example .env          # вписать свой ключ в OPENAI_API_KEY
docker compose up -d
```

Поднимается всё сразу:

| Сервис         | Что это                                        | Порт на хосте |
|----------------|------------------------------------------------|---------------|
| `app`          | сам ассистент (FastAPI)                        | 18000         |
| `bootstrap`    | разовая инициализация: самодокументация, стартовый FAQ, демо-пользователи | — |
| `postgres`     | основное хранилище (пользователи, база знаний) | 15432         |
| `qdrant`       | векторный индекс (производный от Postgres)     | 16333         |
| `redis`        | сессии и ожидающие подтверждения               | 16379         |
| `demo-backend` | фейковый «бэкенд продукта», чтобы команды реально выполнялись | 18100 |

Проверка:

```bash
curl -s -X POST http://localhost:18000/chat \
  -H 'Content-Type: application/json' \
  -d '{"user_id":"acc1","text":"как сбросить пароль?"}'
```

Полезное:

```bash
docker compose logs -f app          # смотреть логи ассистента
docker compose restart app          # перечитать .env / config после правок
docker compose run --rm test        # прогнать тесты (профиль dev)
docker compose down                 # остановить
docker compose down -v              # остановить и стереть данные (Postgres, Qdrant, веса моделей)
```

Первый запуск дольше: качаются образы и веса моделей. Веса лежат в отдельном
томе `modelcache` и переживают пересборку — второй запуск быстрый.

> `bootstrap` индексирует стартовый FAQ, а это значит — считает эмбеддинги. С
> облачной моделью по умолчанию для этого нужен рабочий ключ: без него шаг
> упадёт с `AuthenticationError`. Либо впишите ключ в `.env`, либо переключитесь
> на локальную модель эмбеддингов (раздел 4).

## 3. Подключение модели

Модель нужна классификатору: он превращает фразу «смени мне email и покажи, как
оформить возврат» в список действий. По умолчанию это OpenAI `gpt-5.6-terra` —
достаточно строки `OPENAI_API_KEY=sk-...` в `.env`. **Без доступной модели onbo
не падает**, а переключается на эвристику по ключевым словам — простые команды
работают, свободные формулировки нет.

Смена модели — один блок в `.env`. Правило одно:

> `LLM_API_BASE` задаётся **только** для своего сервера (Ollama, vLLM и т. п.).
> Для облачных провайдеров (OpenAI, Anthropic, Gemini) переменную оставляют
> пустой, иначе запросы уйдут не туда.

### Флагманские облачные модели

Ключ кладётся в один и тот же `LLM_API_KEY` (или в переменную самого провайдера,
например `OPENAI_API_KEY`), провайдер определяется префиксом модели.
`LLM_API_BASE` не задаём.

```ini
# OpenAI — по умолчанию; достаточно OPENAI_API_KEY=sk-... в .env
LLM_MODEL=gpt-5.6-terra
LLM_API_KEY=sk-...
```

```ini
# Anthropic
LLM_MODEL=anthropic/claude-sonnet-4-5
LLM_API_KEY=sk-ant-...
```

```ini
# Google Gemini
LLM_MODEL=gemini/gemini-2.0-flash
LLM_API_KEY=AIza...
```

Внутри работает LiteLLM, поэтому подходит любая строка модели из его каталога
(Azure OpenAI, Bedrock, Mistral и т. д. — у них свои переменные окружения;
добавьте их в `.env`, они дойдут до контейнера как есть).

### Опенсорс-модели у себя

Ничего не уходит за пределы вашей сети, и нет оплаты за запросы.

**Вариант А — Ollama на хосте (использует вашу видеокарту, рекомендуется):**

```bash
# на хосте
curl -fsSL https://ollama.com/install.sh | sh
ollama pull qwen2.5:7b
```

`.env`:

```ini
LLM_MODEL=ollama_chat/qwen2.5:7b
LLM_API_BASE=http://host.docker.internal:11434
```

Важный момент: по умолчанию Ollama слушает только `127.0.0.1`, и контейнер до
неё не достучится (`Connection refused`). Откройте её для Docker:

```bash
sudo systemctl edit ollama     # добавить: [Service] Environment="OLLAMA_HOST=0.0.0.0"
sudo systemctl restart ollama
```

**Вариант Б — Ollama внутри Docker** (ничего не ставить на хост, но только CPU —
проброс GPU требует `nvidia-container-toolkit`, поэтому ответы медленные):

```bash
docker compose --profile ollama up -d
docker compose exec ollama ollama pull llama3.2:3b
```

`.env`:

```ini
LLM_MODEL=ollama_chat/llama3.2:3b
LLM_API_BASE=http://ollama:11434
```

**Вариант В — любой OpenAI-совместимый сервер**: vLLM, LM Studio, llama.cpp
(`llama-server`), OpenRouter, корпоративный прокси:

```ini
LLM_MODEL=openai/Qwen/Qwen2.5-7B-Instruct
LLM_API_BASE=http://host.docker.internal:8000/v1
LLM_API_KEY=not-needed
```

Про размер модели: классификатор просит модель вернуть строгий JSON. Модели
уровня 3B часто отдают его криво — onbo это переживает (падает в эвристику), но
качество разбора фраз заметно хуже. Практический минимум — 7–8B (`qwen2.5:7b`,
`llama3.1:8b`), лучше — instruct-модели с хорошим следованием формату.

После правки `.env`:

```bash
docker compose restart app
```

### Проверить, что модель действительно отвечает

Задайте фразу, которую эвристика не разберёт:

```bash
curl -s -X POST http://localhost:18000/chat -H 'Content-Type: application/json' \
  -d '{"user_id":"acc1","text":"переключи интерфейс на английский и заодно смени почту на a@b.com"}'
```

С работающей моделью в ответе будут **два** результата (смена языка + запрос
подтверждения на email). Если модель недоступна, в логах `app` это видно как
падение на классификации, а ответ окажется беднее.

## 4. Эмбеддинги (поиск по базе знаний)

Модель эмбеддингов превращает текст в вектор, по векторам работает поиск. Это
отдельный от чат-модели выбор — провайдеры могут быть разные.

**Флагманская облачная модель (по умолчанию).** Из коробки это OpenAI
`text-embedding-3-large` (3072 измерения), покрывается тем же
`OPENAI_API_KEY`. Для другого провайдера задаёте модель и ключ — провайдер
определяется по строке модели, ровно как у чат-модели:

| Провайдер | `EMBED_MODEL` | Размер вектора |
|---|---|---|
| OpenAI | `text-embedding-3-small` / `text-embedding-3-large` | 1536 / 3072 |
| Google Gemini | `gemini/gemini-embedding-001` | 3072 |
| Anthropic → Voyage AI | `voyage/voyage-3-large`, `voyage/voyage-3.5-lite` | 1024 |
| Cohere, Mistral, Bedrock, Azure… | напр. `cohere/embed-multilingual-v3.0` | по провайдеру |

```ini
EMBED_MODEL=text-embedding-3-large
EMBED_API_KEY=sk-...
```

Своих эмбеддингов у Anthropic нет, она рекомендует Voyage AI — поэтому в строке
«Anthropic» стоит Voyage, ключ берётся на voyageai.com.

**Опенсорс у себя на машине.** Работает через `fastembed`: ключи и внешние
сервисы не нужны, данные компании никуда не уходят. Лучший мультиязычный
вариант — `intfloat/multilingual-e5-large` (1024 измерения), полегче:

```ini
EMBED_MODEL=intfloat/multilingual-e5-large                                 # 1024d
EMBED_MODEL=sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2   # 384d
```

Правило то же, что с чат-моделью: **`EMBED_API_BASE` — только для своего
сервера** (vLLM, LM Studio, прокси), для облаков его не задают.

`EMBED_PROVIDER=auto` (по умолчанию) сам понимает, локальная модель или
облачная: имя без слэша (`text-embedding-3-small`) или известный префикс
провайдера — значит API, а строка вида `org/model` — значит fastembed. Если имя
ломает это правило, задайте явно `api` или `local` — например для открытой
модели на своём vLLM:

```ini
EMBED_PROVIDER=api
EMBED_MODEL=Qwen/Qwen3-Embedding-0.6B
EMBED_API_BASE=http://host.docker.internal:8000/v1
EMBED_API_KEY=not-needed
```

Чем платите: с облачной моделью текст базы знаний уходит провайдеру — один раз
при индексации и по разу на каждый вопрос. Локальная не отправляет никуда
ничего, но чуть хуже на редких формулировках. Качество влияет только на поиск:
сам текст ответа — тот, который вы написали в базу.

Любая смена модели меняет размер вектора, индекс надо пересобрать:

```bash
docker compose exec app onbo kb reindex
```

## 5. Голосовой ввод

Распознавание речи — `faster-whisper`, тоже локально. В контейнере GPU нет,
поэтому compose принудительно ставит `STT_DEVICE=cpu`, `STT_COMPUTE=int8` и
модель `base`. Крупнее — точнее, но на CPU заметно медленнее:

```ini
STT_MODEL=small     # base | small | medium | large-v3
STT_ENABLED=true
```

Как включить микрофон в вебе — во втором файле, раздел про чат.

## 6. Связка со своим продуктом

Ассистент не только отвечает, но и выполняет команды — дёргает HTTP API вашего
продукта. Пока `PRODUCT_API_BASE` пуст, действия работают в режиме сухого
прогона: onbo проверит параметры и скажет, что *собирался* вызвать, но запроса
не сделает. В Docker по умолчанию подставлен встроенный демо-бэкенд, чтобы
всё было видно вживую.

`.env`:

```ini
PRODUCT_NAME=Acme CRM
PRODUCT_DESCRIPTION=CRM для отдела продаж
PRODUCT_API_BASE=https://api.acme.example.com
PRODUCT_API_KEY=...              # уйдёт в заголовок Authorization: Bearer ...
PRODUCT_AUTH_HEADER=Authorization
PRODUCT_AUTH_SCHEME=Bearer       # пусто = передавать ключ как есть, без префикса
```

Сами команды описываются в `config/actions.yaml` — это тема второго файла.

## 7. Пользователи, отделы и роли

Профиль пользователя (`department`, `roles`) — единственный источник фильтра
доступа: он решает, какие материалы попадут в ответ. Он берётся **из базы**, а
не из текста сообщения и не от модели.

- Порядок поиска: таблица `app_user` в Postgres → встроенный демо-справочник
  (`acc1`, `sup1`, `admin`) → неизвестный пользователь получает минимальные
  права (только публичные материалы и раздел «о себе»).
- Для своего проекта заполните `app_user` (`user_id`, `department`, `roles`) из
  своей кадровой системы/каталога — синхронизацией на своей стороне.

Посмотреть демо-справочник:

```bash
docker compose exec postgres psql -U onbo -d onbo -c 'select * from app_user'
```

## 8. Что можно выключить

Каждый флаг убирает целую подсистему вместе с её HTTP-ручками:

```ini
FEATURE_CHAT=true          # /chat, /voice, /confirm
FEATURE_ADMIN=true         # /admin — панель базы знаний
FEATURE_MEDIA=true         # /media — видео-ролики
FEATURE_LLM_MANIFEST=true  # /llm.json для внешних ИИ-агентов
FEATURE_WELCOME=true       # проактивное приветствие
FEATURE_ACTIONS=true       # выполнение команд
FEATURE_RAG=true           # ответы из базы знаний
```

Например, «только команды, без базы знаний» — это `FEATURE_RAG=false`.

## 9. Перед показом живым людям

- **`/admin` открыт без пароля.** Кто угодно с доступом к порту 18000 правит базу
  знаний. Наружу выставлять нельзя — только за VPN/прокси с авторизацией.
  Аутентификация админки записана в бэклог (`.claude/PLAN3.md`, пункт 20).
- **`user_id` приходит из запроса и никак не проверяется.** Для реального
  внедрения поставьте перед onbo свой прокси, который проверяет сессию
  сотрудника и сам подставляет `user_id`.
- Демо-пользователей и демо-бэкенд из прод-конфига уберите
  (`PRODUCT_API_BASE` — на реальный API).
- Пароли Postgres в `docker-compose.yml` — дефолтные (`onbo/onbo`).

## 10. Если что-то не так

| Симптом | Причина и что делать |
|---|---|
| Ответы примитивные, сложные фразы не понимаются | Модель недоступна, работает эвристика. Проверьте `docker compose logs app`, `LLM_MODEL`/`LLM_API_KEY`, и что `LLM_API_BASE` **не** задан для облака |
| `Connection refused` к Ollama | Ollama слушает `127.0.0.1`: включите `OLLAMA_HOST=0.0.0.0` и перезапустите её |
| `localhost:18000` не открывается | `docker compose ps` — жив ли `app`; `bootstrap` должен завершиться успешно, `app` ждёт его |
| Первый запрос очень долгий | Качаются веса эмбеддингов/whisper. Дальше они берутся из тома `modelcache` |
| Поиск ничего не находит после смены `EMBED_MODEL` | `docker compose exec app onbo kb reindex` |
| `AuthenticationError` при индексации или на вопросе | Облачная модель эмбеддингов без ключа или с чужим. Ключ отдельный на провайдера: `EMBED_API_KEY` ≠ `LLM_API_KEY` |
| Порт занят | Хостовые порты сдвинуты на +10000 (15432/16333/16379); веб-порт меняется через `WEB_PORT` |
| Хочется начать с чистого листа | `docker compose down -v && docker compose up -d` |
